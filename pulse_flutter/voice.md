Клиент поддерживает работу по двум сетевым протоколам с автоматическим резервированием.
Вариант А: TCP (WebSocket) — Резервный канал
Порт: 8765 (по умолчанию)
URL-схема: ws://<HOST>:8765/ws?room=<ROOM_ID>&nick=<NICKNAME> (или wss:// при работе через HTTPS)
Формат фреймов: Строго бинарные кадры (binary frame).
Фрейминг пакетов: Не требуется на уровне приложения (WebSocket обеспечивает внутреннее фрейминг-разделение сообщений).
Вариант Б: UDP (WebTransport over HTTP/3) — Основной канал
Порт: 4433 (по умолчанию)
URL-схема: https://<HOST>:4433/wt?room=<ROOM_ID>&nick=<NICKNAME>
Медиа-трафик (Пакеты типа 0x01): Передаются и принимаются через ненадежный канал датаграмм (Datagrams). При потере пакета повторная отправка не происходит.
Сигнальный трафик (Пакеты типов 0x02 - 0x07): Передаются и принимаются через надежные однонаправленные потоки (Unidirectional Streams).
Фрейминг потоков (WT Stream Framing): Поскольку поток данных внутри QUIC Stream представляет собой непрерывный массив байтов, все пакеты, отправляемые или принимаемые через Stream, снабжаются префиксом длины:
[Длина сообщения (4 байта, Uint32, Big-Endian)] + [Тело пакета]
2. Спецификация бинарного протокола (Форматы пакетов)
Все данные передаются в виде Uint8Array. Первый байт (индекс 0) строго определяет тип пакета.
Все числовые идентификаторы (Sender_ID, Peer_ID, Target_ID) представляют собой 32-битные целые числа без знака (Uint32, 4 байта, Big-Endian).
[16.07.2026 20:22] sanlsan: 3. Криптографический протокол E2EE (Алгоритм)
Шифрование реализовано по схеме Sender Keys (Ключи отправителя) с использованием встроенного браузерного WebCrypto API:
ECDH (Curve P-256) — используется асимметричная пара ключей для безопасного согласования общих секретов «точка-точка» между участниками.
AES-GCM (256-bit) — симметричный алгоритм для шифрования медиаданных и обмена ключами. Требует уникальный IV (12 байт) для каждой операции шифрования.
Пошаговый жизненный цикл сессии клиента
Шаг 1. Локальная инициализация
Сразу после нажатия кнопки «Войти» клиент генерирует:
Асимметричную пару ECDH P-256 (myKeyPair).
Случайный симметричный ключ AES-GCM 256-bit (mySenderKey). Это постоянный ключ, которым клиент будет шифровать все свои исходящие медиапакеты.
Экспортирует свой публичный ключ (myKeyPair.publicKey) в формат raw (65 байт) ➔ myPubKeyRaw.
Экспортирует свой медиа-ключ (mySenderKey) в формат raw (32 байта) ➔ mySenderKeyRaw.
Шаг 2. Вход и объявление себя (Пакет 0x05)
После успешного подключения транспортного уровня клиент получает пакет 0x03 с назначенным ему My_ID.
Клиент отправляет пакет публичного ключа: [0x05] + [myPubKeyRaw] (66 байт).
(Сервер начнет пропускать медиа-трафик клиента только после того, как получит от него публичный ключ).
Шаг 3. Обработка новых участников и согласование общих секретов
При получении пакета 0x05 от сервера ([0x05] + [Peer_ID (4b)] + [Peer_PubKey (65b)]):
Клиент импортирует Peer_PubKey через crypto.subtle.importKey.
Клиент вычисляет общий секретный ключ sharedAesKey (AES-GCM 256) путем вызова crypto.subtle.deriveKey на основе myKeyPair.privateKey и импортированного Peer_PubKey.
Генерирует случайный IV (12 байт).
Шифрует свой экспортированный ранее mySenderKeyRaw (32 байта) на ключе sharedAesKey с использованием сгенерированного IV. Получает EncryptedKey (размер ровно 48 байт: 32 байта данных + 16 байт аутентификационного тега AES-GCM).
Отправляет пакет 0x06 на сервер для передачи ключа адресату:
[0x06] + [Peer_ID (4b)] + [IV (12b)] + [EncryptedKey (48b)] (65 байт).
Клиент пересчитывает код сессии (см. раздел 4).
Шаг 4. Прием чужих медиа-ключей
При получении пакета 0x06 от сервера ([0x06] + [Sender_ID (4b)] + [IV (12b)] + [EncryptedKey (48b)]):
Клиент берет sharedAesKey для этого Sender_ID (вычисленный на Шаге 3).
Дешифрует EncryptedKey с использованием IV и sharedAesKey.
Импортирует полученные 32 байта расшифрованного ключа как симметричный ключ AES-GCM (peerSenderKeys[Sender_ID]). Теперь клиент может расшифровывать медиапоток от этого участника.
Шаг 5. Шифрование и дешифрование медиа
Отправка: Перед отправкой Opus-кадра клиент генерирует случайный IV (12 байт), шифрует кадр своим mySenderKey и упаковывает в 0x01 пакет: [0x01] + [IV (12b)] + [Ciphertext].
Прием: При получении пакета 0x01 ([0x01] + [Sender_ID (4b)] + [IV (12b)] + [Ciphertext]): клиент извлекает Sender_ID, берет из словаря peerSenderKeys[Sender_ID] и дешифрует кадр с помощью пришедшего IV.
4. Верификационный код сессии (4 эмодзи)
Для гарантированной защиты от атаки «Человек посередине» (MITM) на этапе обмена ключами ECDH, в интерфейс выводится код сессии. Метод расчета кода детерминирован и одинаков для всех клиентов в комнате:
Формируется массив участников комнаты (включая себя). Каждый элемент содержит ID (4 байта) и PubKey_Raw (65 байт).
Массив строго сортируется по возрастанию ID участников.
Все элементы конкатенируются в один непрерывный буфер байтов. Размер буфера: N * 69 байт.
Вычисляется криптографический хэш SHA-256 от этого буфера.
Берутся первые 4 байта полученного хэша.
Каждый байт проецируется на массив из 64 визуально отличимых эмодзи методом остатка от деления: emojiList[byte % 64]. Получаются 4 эмодзи.
